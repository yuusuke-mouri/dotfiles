# 実装型コンサルタントが出来ること ─ エンジニアの技術を活かしたコスパDX

## 「実装型」の3つの意味を深堀り

Day15で、「実装型コンサルタント」には3つの意味があると書きました。

  1. **作れる** ─ 自分で手を動かせる

  2. **分かる** ─ 技術的な実現可能性を判断できる

  3. **伝えられる** ─ 開発会社に明確に指示できる

Day16-17では「枯れた技術」「ポータブルな技術」について書きました。  
今日は「分かる」をもう少し具体的に深掘りします。

Day8で「二つの世界線」を書きました。

世界線2（フィクション）では、私はこんな判断をしました。

> \- Shopify → API連携OK  
> \- 楽天 → RMS API OK  
> \- Amazon → MWS API OK（認証面倒だが可能）  
> \- ZOZOTOWN → APIなし、CSV手動でOK  
> \- POS（スマレジ）→ API連携OK

なぜこの判断と提案ができたのか。

## API ─ 枯れた技術

API（Application Programming Interface）は、システム同士がデータをやり取りする仕組みです。

これは何十年も使われてきた「枯れた技術」であり、**システム開発の世界で標準化された仕組み** です。

車で例えると、**運転操作の標準化** のようなものです。  
どのメーカーの車でも、アクセル・ブレーキ・ハンドルの基本操作は同じですよね。  
トヨタからホンダに乗り換えても、運転の仕方を一から覚え直す必要はありません。

APIも同じです。Shopifyでも楽天でもAmazonでも、APIという共通の考え方でデータをやり取りできます。  
（注意：各システムがAPI対応している必要あり）  
ツールが変わっても使える、まさに「ポータブルな技術」です。

  * Shopifyには商品・注文データを取得するAPIがある

  * 楽天には RMS API がある

  * Amazonには MWS API がある

  * スマレジにもAPIがある

私はこの「APIがあるかどうか」を知っている、もしくは調査するスキルがあります。  
だから「API を使って自動化できるか、それとも手動や別の工夫が必要になるか」を判断できました。

APIという概念を知らなければ、この判断はできません。  
「全部自動化したい」と言って、開発会社に丸投げしていたかもしれません。

## GCP ─ グローバルで安定したインフラ

一方、GCP（Google Cloud Platform）は**Googleが提供するクラウドインフラサービス** です。  
知名度の高いAWS（Amazon Web Services）と類似したサービス（IaaS: Infrastructure as a Service）で、ここ10年ほどで普及しました。

車で例えると、**カーリース会社** のようなものです。  
自分で車を買わなくても、必要な時に必要な分だけ借りられる。  
GCPは「Googleのカーリース」、AWSは「Amazonのカーリース」というイメージです。

  * Cloud Functions：サーバーレスでコードを実行

  * BigQuery：大規模データを高速に分析

  * Looker Studio：データをダッシュボード化

APIが「運転操作の標準化」なのに対し、GCPは「カーリース会社」という一製品です。  
でも、Googleがグローバルに提供し、ビジネスインフラとしての地位を確立しています。  
Day17で書いた「ポータブルな技術」の延長線上にあります。

## 枯れた技術と新しいインフラ、両方を扱う

私はスペシャリストではないですが、複雑なシステム開発でなければ両方を扱えます。

**枯れた技術** :

  * API、SQL、JavaScript

  * 何十年も使われてきた基礎

**新しいインフラ** :

  * GCP、AWS、Azure

  * グローバルで安定、情報も豊富

枯れた技術で基礎を固めつつ、新しいインフラにキャッチアップする。  
これが「分かる」の背景にあるスキルセットです。

## なぜキャッチアップできるのか

「新しい技術にキャッチアップし続けるのは大変では？」

確かに大変です。  
でも、運転の基本を知っていれば、新しい車種でもすぐ運転できますよね。  
ガソリン車を運転できる人なら、電気自動車でもハイブリッド車でも、基本操作は同じです。

技術も同じです。基礎があれば、新しいサービスでも理解が早い。  
「ポータブルな技術」の強みです。

## 経営者には見えにくい部分

技術的な判断は、経営者には見えにくい部分です。

「ECサイトの売上を分析したい」という要望に対して、「Shopifyは自動、ZOZOTOWNは手動」という判断が必要だとは、普通は思いません。

でも、この判断をしないまま進めると、Day8の世界線1のように「実装フェーズで問題噴出」となります。

> \- 開発会社：「楽天は仕様が特殊です」  
> \- 開発会社：「ZOZOはAPI制限があります」  
> \- 経営者：「そんなこと最初に確認してよ」  
> \- 追加費用：200万円、納期延長：3ヶ月

[経営者「ECサイトとPOSの売上を分析できるようにしよう」から始まる二つの世界線](https://note.com/yusukemori_ravi/n/n84016de149a2)

「最初に確認する」ができるのは、技術を分かっているからです。

## まとめ

「分かる」とは、技術的な実現可能性を判断できること。

  * API（枯れた技術）で「自動化できるか」を判断

  * GCP（新しいインフラ）で「どう実装するか」を提案

  * 枯れた技術の基礎があるから、新しい技術にキャッチアップできる

これが「実装型コンサルタント」の「分かる」という強みです。
