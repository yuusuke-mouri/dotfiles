# システム開発やDXの複雑性への対策まとめ(Week 2 まとめ)

## Week2 を振り返る

今週は Day8 の「なぜソフトウェア開発は難しいのか」という問いから始まり、その解決策を順番に紹介してきました。

[](https://note.com/yusukemori_ravi/n/n84016de149a2)

[](https://note.com/yusukemori_ravi/n/nccb606b69dab)

この流れを踏まえて、今日は Day8 で紹介した「二つの世界線」を改めて振り返ってみます。

## 世界線2 はなぜ成功したのか

世界線2 では、2ヶ月で Phase 1 が完成し、経営者から「完璧！」という言葉をもらいました。  
この話はフィクションですが、私の経験では似たようなことは起きています。

世界線2 が成功した理由を見てみましょう。

**1\. 業務の複雑性を受け入れた**

世界線2では「最初から完璧な要件定義は無理」という前提で始めました。  
経営者の頭の中にある「曖昧な仏像」を、一度に全部彫ろうとしなかった。

**2\. 小さく作った（MVP）**

Phase 1（2ヶ月）では「見る」機能だけに絞りました。  
「移動指示」は Phase 2 に回すことで、早く価値を届けられました。

[](https://note.com/yusukemori_ravi/n/n07415e55f3f1)

**3\. 途中で確認した（アジャイル）**

2週間ごとに経営者に見せて、フィードバックをもらいました。  
「商品別にも見たい」「在庫との比較も欲しい」という声を、次のスプリントに反映できました。

[](https://note.com/yusukemori_ravi/n/n61569bb063ca)

**4\. 頭の中を言語化した（ユーザーストーリー）**

「在庫配分を最適化したい」という曖昧な要望を、ユーザーストーリーに分解しました。  
「誰が」「何を」「なぜ」を明確にすることで、開発チームも迷わず実装できました。

[](https://note.com/yusukemori_ravi/n/nb54a99792d78)

**5\. 経営者と対話した**

戦略実行設計者が経営者と対話し、意図を確認しながら進めました。  
営業部の佐藤さんにも確認してもらい、現場の声も反映できました。

## 世界線1 はなぜ失敗したのか

一方、世界線1 では 1年かけて開発したシステムが「これじゃない」と言われました。

**1\. 業務の複雑性を無視した**

曖昧な「やりたいこと」を伝えて、 部下に依頼しました。  
「分かるだろう」と思ったのかもしれません。

**2\. 一度に全部作った**

6〜12ヶ月かけて、すべての機能を一度に開発しました。  
その間、経営者は完成品を見ることができませんでした。

**3\. 途中で確認しなかった**

完成後に初めて経営者に見せたら、「これじゃない」と言われました。

**4\. 言語化しなかった**

「いい感じにして」という曖昧な指示のまま、開発が進みました。

**5\. 対話が少なかった**

IT 部門と経営者の間に断絶がありました。

## まとめ

違いはシンプルです。

> **「一度で完璧」を目指すか、「小さく作って確認」を繰り返すか。**

Week2 では、ソフトウェア開発が難しい理由（業務の複雑性）と、その解決策（MVP、アジャイル、ユーザーストーリー）を紹介しました。  
私は前職でウォーターフォールと呼ばれる開発プロセスしか経験してなかったのですが、当時からアジャイルの勉強を始めていて、いまは「アジャイル」をキーワードとする現場が増えてきたこともあり「この考え方じゃないとシステム開発うまくいかない」と思えるようになりました。ただし、言うは易し。実践は難しいです。

さて、来週（Week3）は「実装型コンサルタント」を名乗る理由を深掘りしていきます。

Day8 の世界線2 で、私（戦略実行設計者）が何をしていたか。  
コンテキストの理解、技術的判断、開発指示、品質確認など、何が「実践型」なのか具体的に書いていきます。
